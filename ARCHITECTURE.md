# Harness ŌĆö Architecture Guide

Harness is an AI chat application that lets users create configurable "harnesses" ŌĆö named presets pairing an LLM model with one or more MCP (Model Context Protocol) tool servers ŌĆö and then have streaming conversations where the LLM can call tools on those servers in an agentic loop.

## High-Level Architecture

The system is a **Turborepo monorepo** with three packages:

| Package | Tech | Purpose |
|---|---|---|
| `apps/web` | React 19, TanStack Start/Router, Clerk, Convex, Tailwind v4 | The frontend SPA ŌĆö all UI, routing, and real-time data subscriptions |
| `packages/convex-backend` | Convex | The database and serverless backend ŌĆö schema, queries, and mutations for all persistent data |
| `packages/fastapi` | Python, FastAPI, httpx | The AI inference service ŌĆö streams LLM responses via OpenRouter, executes MCP tool calls, handles MCP OAuth |

Data flows in a triangle:

```
        ŌöīŌöĆŌöĆŌöĆ Convex React Client (real-time queries/mutations) ŌöĆŌöĆŌöĆŌöÉ
        Ōöé                                                         Ōöé
    apps/web                                              convex-backend
        Ōöé                                                         Ōöé
        ŌööŌöĆŌöĆ SSE stream (POST /api/chat/stream) ŌöĆŌöĆŌ¢║ fastapi ŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöś
                                                  (saves messages
                                                   via HTTP API)
```

The **frontend** reads/writes harnesses, conversations, and messages through the Convex React client (real-time subscriptions). When the user sends a chat message, the frontend opens an SSE stream to **FastAPI**. FastAPI calls **OpenRouter** for LLM completions, executes any tool calls on MCP servers, and when the full response is done, saves the assistant message to **Convex** via its HTTP API using a deploy key.

## `packages/convex-backend` ŌĆö Database

### Schema (`convex/schema.ts`)

Defines five tables:

| Table | Key Fields | Purpose |
|---|---|---|
| `harnesses` | `name`, `model`, `status`, `mcpServers[]`, `skills[]`, `suggestedPrompts[]`, `userId` | User-created AI presets. Each harness pairs a model with MCP servers. Indexed by `userId`. |
| `conversations` | `title`, `lastHarnessId`, `userId`, `lastMessageAt` | Chat sessions. Each belongs to a user and tracks which harness was last used. Indexed by `userId` and `lastMessageAt` (for sorted listing). |
| `messages` | `conversationId`, `role`, `content`, `reasoning`, `toolCalls[]`, `parts[]`, `usage`, `model`, `interrupted` | Individual messages in a conversation. Supports structured parts (text/reasoning/tool_call) for rich rendering. Indexed by `conversationId`. |
| `mcpOAuthTokens` | `userId`, `mcpServerUrl`, `accessToken`, `refreshToken`, `expiresAt`, `scopes`, `authServerUrl` | Stored OAuth tokens for MCP servers that require OAuth authentication. Indexed by `(userId, mcpServerUrl)`. |
| `userSettings` | `userId`, `autoSwitchHarness`, `displayMode` | Per-user preferences. `displayMode` controls message detail level (zen/standard/developer). |

### Convex Modules

Each module defines queries and mutations that are called from the frontend (via `api.*`) or from FastAPI (via the HTTP API with a deploy key).

| File | Exports | Description |
|---|---|---|
| `convex/auth.config.ts` | default config | Configures Clerk as the JWT auth provider using the `CLERK_JWT_ISSUER_DOMAIN` env var. |
| `convex/harnesses.ts` | `list`, `get`, `create`, `update`, `remove` | CRUD for harnesses. All operations are auth-gated ŌĆö users can only access their own harnesses. `create` auto-sets `userId` and `lastUsedAt`. |
| `convex/conversations.ts` | `list`, `get`, `create`, `updateTitle`, `remove` | CRUD for conversations. `list` returns the 50 most recent by `lastMessageAt`. `remove` cascade-deletes all messages in the conversation. `create` requires a valid `harnessId`. |

| `convex/userSettings.ts` | `get`, `update` | User preferences. `get` returns defaults if no record exists. `update` upserts. |
| `convex/seed.ts` | `seedAll`, `clearAll`, `clearAndReseed` | Dev-only data seeding. Creates sample harnesses and conversations for testing. |
| `convex/migrations.ts` | ŌĆö | Placeholder for future data migrations (currently empty). |

| `app/dependencies.py` | FastAPI dependency injection helpers. `get_http_client` provides the shared httpx client from `app.state`. `get_current_user` wraps `verify_token` for auth. |

### Routes

| File | Prefix | Endpoints | Description |
|---|---|---|---|
| `app/routes/health.py` | `/` | `GET /health` | Simple health check that returns `{"status": "ok"}`. |
| `app/routes/chat.py` | `/api/chat` | `POST /stream` | The core chat endpoint. Accepts a `ChatRequest`, fetches MCP tools via `list_tools`, then enters an agentic loop: streams LLM responses from OpenRouter via `stream_chat`, accumulates tool calls, executes them in parallel via `call_tool`, feeds results back, and repeats (up to 50 iterations). Emits SSE events: `token`, `thinking`, `tool_call`, `tool_result`, `mcp_error`, `done`, `error`. On completion, saves the full message to Convex via `save_assistant_message`. Handles client disconnection gracefully ŌĆö drains the stream to capture usage data and patches it via `patch_message_usage`. |
| `app/routes/mcp_health.py` | `/api/mcp/health` | `POST /check`, `POST /generate-prompts` | `/check` runs health checks on a list of MCP servers in parallel (tries the initialize handshake). `/generate-prompts` fetches available tools from MCP servers and uses a quick LLM call (via `complete_chat`) to generate 4 suggested prompts based on the tools. |
| `app/routes/mcp_oauth.py` | `/api/mcp/oauth` | `GET /start`, `GET /callback`, `GET /status`, `POST /revoke` | OAuth flow for MCP servers. `/start` discovers the auth server, registers a client (via DCR), generates PKCE, and returns an authorization URL. `/callback` exchanges the auth code for tokens and stores them in Convex. `/status` checks if the user has a valid token. `/revoke` deletes stored tokens. |

### Services

| File | Key Functions | Description |
|---|---|---|
| `app/services/openrouter.py` | `stream_chat`, `complete_chat` | OpenRouter API client. `stream_chat` opens a streaming POST to `https://openrouter.ai/api/v1/chat/completions`, resolves model names via `MODEL_MAP`, enables reasoning for thinking models, and yields parsed SSE chunks. `complete_chat` is a non-streaming variant used for lightweight tasks like prompt generation. |
| `app/services/mcp_client.py` | `list_tools`, `call_tool`, `check_server_health`, `evict_session_cache` | MCP server client. Handles the full JSON-RPC 2.0 protocol: session initialization, tool listing (with 60s TTL cache), tool execution, and health checks. Sanitizes tool names into `servername__toolname` format for OpenAI compatibility. Supports both JSON and SSE response modes. Handles OAuth 401s by raising `McpAuthRequiredError`. |
| `app/services/mcp_oauth.py` | `discover_auth_server`, `start_oauth_flow`, `exchange_code`, `refresh_access_token`, `get_valid_token` | Full MCP OAuth 2.1 implementation. Discovers authorization servers via Protected Resource Metadata (RFC 9728), performs Dynamic Client Registration (RFC 7591), generates PKCE challenges, builds authorization URLs with resource indicators (RFC 8707), exchanges codes for tokens, and auto-refreshes expired tokens. Stores/reads tokens from Convex via HTTP API. |
| `app/services/convex.py` | `save_assistant_message`, `patch_message_usage` | Convex HTTP API client. Uses the deploy key for admin authentication to call internal mutations. `save_assistant_message` persists a complete assistant message (content, reasoning, tool calls, parts, usage, model). `patch_message_usage` backfills usage data on interrupted messages. |

## Key Data Flows

### 1. Sending a Chat Message

1. User types in `chat/index.tsx` ŌåÆ calls `api.messages.send` (Convex mutation) to save the user message
2. `chat/index.tsx` calls `useChatStream.stream()` with the message history + harness config
3. `use-chat-stream.ts` opens a `fetch()` SSE stream to `POST /api/chat/stream` with a Clerk JWT
4. `routes/chat.py` authenticates the JWT via `verify_token`, fetches MCP tools via `list_tools`
5. `routes/chat.py` enters the agentic loop ŌĆö calls `stream_chat` (OpenRouter), processes tool calls via `call_tool`, streams events back
6. On completion, `routes/chat.py` calls `save_assistant_message` (Convex HTTP API) to persist the message
7. `routes/chat.py` sends the `done` SSE event with the final content and usage data
8. Frontend receives `done`, Convex real-time subscription picks up the new message

### 2. MCP OAuth Connection

1. Frontend's `McpServerStatus` detects a server needs OAuth ŌåÆ user clicks "Reconnect"
2. `startOAuthPopup` in `mcp-server-status.tsx` calls `GET /api/mcp/oauth/start?server_url=...` with a Clerk JWT
3. FastAPI's `mcp_oauth.py` route calls `start_oauth_flow` which discovers the auth server, registers a client, generates PKCE, returns an authorization URL
4. Frontend opens the URL in a popup window
5. User authorizes ŌåÆ auth server redirects to `GET /api/mcp/oauth/callback?code=...&state=...`
6. FastAPI exchanges the code for tokens via `exchange_code`, stores them in Convex via `_convex_store_tokens`
7. FastAPI returns HTML that posts a message to the opener window and closes the popup
8. Frontend receives the `postMessage`, shows success toast

### 3. MCP Tool Execution (inside agentic loop)

1. OpenRouter returns `finish_reason: "tool_calls"` with one or more tool call deltas
2. `routes/chat.py` accumulates the deltas, parses arguments, sends `tool_call` SSE events to the frontend
3. All tool calls are executed in parallel via `asyncio.gather` ŌåÆ `call_tool` in `mcp_client.py`
4. `call_tool` resolves the `servername__toolname` format, finds the right MCP server, sends a JSON-RPC `tools/call` request
5. Results are streamed back as `tool_result` SSE events
6. Tool results are appended to the message history, and the loop continues with another OpenRouter call

## Getting Started

```bash
# Install dependencies
bun install

# Start all three services in dev mode
bun run dev
# This runs (via turbo):
#   apps/web        ŌåÆ http://localhost:3000
#   convex-backend  ŌåÆ Convex dev server
#   fastapi         ŌåÆ http://localhost:8000

# Or start individually:
cd apps/web && bun run dev
cd packages/convex-backend && bun run dev
cd packages/fastapi && bun run dev
```

### Required Environment Variables

**`apps/web/.env.local`:**
- `VITE_CONVEX_URL` ŌĆö Your Convex deployment URL
- `VITE_CLERK_PUBLISHABLE_KEY` ŌĆö Clerk publishable key
- `VITE_FASTAPI_URL` ŌĆö FastAPI URL (defaults to `http://localhost:8000`)

**`packages/convex-backend/.env.local`:**
- `CLERK_JWT_ISSUER_DOMAIN` ŌĆö Your Clerk JWT issuer domain

**`packages/fastapi/.env`:**
- `OPENROUTER_API_KEY` ŌĆö OpenRouter API key (required)
- `CONVEX_URL` ŌĆö Convex deployment URL
- `CONVEX_DEPLOY_KEY` ŌĆö Convex deploy key (for admin HTTP API access)
- `FRONTEND_URL` ŌĆö Frontend URL for CORS (defaults to `http://localhost:3000`)
- `GITHUB_OAUTH_CLIENT_ID` / `GITHUB_OAUTH_CLIENT_SECRET` ŌĆö Optional, for GitHub MCP OAuth
